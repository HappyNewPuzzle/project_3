# 서버 고도화 로드맵

이 문서는 서버 MVP 이후 실제 운영 게임 서버 수준으로 확장하기 위해 진행할 작업 목록을 정리합니다.

현재 서버 MVP는 방치형 게임의 핵심 루프와 서버 권위 구조를 검증하는 데 초점을 맞췄습니다. 서버 고도화 단계는 여기서 한 걸음 더 나아가 많은 유저, 악용 방지, 운영 대응, 배포 자동화, 장기 데이터 관리를 다루는 단계입니다.

## 1. 고도화 목표

서버 고도화의 목표는 다음입니다.

- 재화와 성장 이력을 운영 중 추적할 수 있게 한다.
- 비정상 요청과 과도한 호출을 제한한다.
- 운영자가 유저 상태를 확인하고 문제를 대응할 수 있게 한다.
- 배포와 설정을 재현 가능하게 만든다.
- 이후 Redis, 장비/스킬, 상점, 랭킹 같은 확장을 안전하게 붙일 기반을 만든다.

## 2. 진행 원칙

- 서버 MVP의 핵심 API 계약은 깨지지 않게 유지합니다.
- 재화와 성장 결과의 원천 데이터는 PostgreSQL에 영속 저장합니다.
- Redis는 필요가 생긴 기능에 한해 보조 저장소로 도입합니다.
- 결제는 실제 결제보다 모의 상점/구매 이력부터 시작합니다.
- 각 고도화 Step은 구현, 테스트, 문서, Git 커밋과 푸시까지 완료해야 종료합니다.

## 3. 추천 진행 순서

### Hardening Step 1. 재화 변경 이력 원장

상태: 완료

#### 왜 먼저 하나?

운영 게임 서버에서 가장 중요한 데이터는 재화입니다. 유저가 골드를 왜 얻었고 왜 썼는지 추적할 수 있어야 버그, 악용, CS 문의에 대응할 수 있습니다.

#### 할 일

- `gold_ledger_entries` 같은 재화 변경 이력 테이블 추가
- 보상 수령, 영웅 강화, 스테이지 체크포인트 정산에서 이력 기록
- 변경 전/후 골드, 변경량, 사유, 참조 키 저장
- 재화 이력 조회용 Application/API 추가 여부 검토

구현 결과:

- 실제 골드가 변한 경우에만 `gold_ledger_entries`에 기록합니다.
- 게임 상태, 기능 영수증, 원장을 한 번의 저장으로 원자적으로 반영합니다.
- `(player_id, reason, reference_id)` 유일 제약으로 중복 원장을 차단합니다.
- 일반 클라이언트 조회 API는 노출하지 않고 관리자 조회 Step에서 추가하기로 결정했습니다.
- 상세 규칙은 [골드 변경 이력 원장](GOLD_LEDGER.md)에 정리했습니다.

#### 완료 기준

- 모든 골드 증가/감소가 원장에 기록됩니다.
- 기존 골드 상태와 원장 기록이 테스트로 함께 검증됩니다.
- 동시 요청에서도 중복 원장 기록이 생기지 않습니다.

### Hardening Step 2. Rate Limiting

상태: 완료

#### 왜 필요한가?

멱등성은 중복 지급을 막지만, 과도한 요청 자체를 막지는 않습니다. 운영에서는 유저별/IP별 요청 제한이 필요합니다.

#### 할 일

- ASP.NET Core Rate Limiting 적용
- 인증 API와 상태 변경 API에 다른 제한 정책 적용
- 429 오류 계약 문서화
- 테스트에서 제한 초과 응답 검증

구현 결과:

- 게스트 생성은 연결 IP당 5회/분으로 제한합니다.
- 보상·강화·스테이지 변경은 JWT 플레이어당 30회/분으로 제한합니다.
- 제한 초과는 `Retry-After`가 포함된 429 `ProblemDetails`로 통일했습니다.
- 같은 IP의 서로 다른 플레이어가 독립된 변경 한도를 사용하는지 API 테스트로 검증했습니다.
- 단일 프로세스 메모리 카운터와 다중 인스턴스 확장 조건을 [API 요청 속도 제한](RATE_LIMITING.md)에 기록했습니다.

#### 완료 기준

- 과도한 요청이 429로 제한됩니다.
- 정상적인 게임 플레이 흐름은 제한에 걸리지 않습니다.
- 오류 계약 문서에 429 처리 방식이 추가됩니다.

### Hardening Step 3. 관리자용 조회 API

상태: 완료

#### 왜 필요한가?

운영 중에는 특정 유저의 상태와 재화 이력을 확인해야 합니다. 관리자 도구 전체를 만들기 전, 서버 API 수준에서 읽기 전용 조회부터 준비하는 것이 좋습니다.

#### 할 일

- 관리자 인증 방식 설계
- 유저 게임 상태 조회 API
- 유저 재화 이력 조회 API
- 관리자 API를 일반 클라이언트 API와 경로/권한으로 분리

구현 결과:

- 서명된 JWT의 `account_type=admin`을 요구하는 관리자 전용 Policy를 추가했습니다.
- 특정 플레이어의 서버 상태를 읽는 API를 추가했습니다.
- 골드 원장을 최대 100개씩 최신순 커서 페이지로 읽는 API를 추가했습니다.
- 게스트 403, 관리자 성공, 잘못된 커서 400, PostgreSQL 키셋 조회를 테스트했습니다.
- 운영 인증과 감사 로그의 현재 경계를 [관리자 조회 API](ADMIN_API.md)에 기록했습니다.

#### 완료 기준

- 관리자 권한 없이는 접근할 수 없습니다.
- 특정 플레이어 상태와 이력을 조회할 수 있습니다.
- 민감한 운영 API가 OpenAPI/문서에서 명확히 구분됩니다.

### Hardening Step 4. Dockerfile과 배포 자동화 기초

상태: 완료

#### 왜 필요한가?

현재는 로컬 실행 중심입니다. 운영에 가까워지려면 같은 방식으로 빌드하고 실행할 수 있는 컨테이너 이미지가 필요합니다.

#### 할 일

- API 서버용 Dockerfile 추가
- `.dockerignore` 추가
- 로컬에서 이미지 빌드 검증
- Production 환경 변수 주입 방식 문서화
- Migration 적용 절차와 API 실행 절차 분리

구현 결과:

- .NET SDK와 ASP.NET Runtime을 분리한 Alpine 다단계 `Dockerfile`을 추가했습니다.
- `.dockerignore`로 비밀값, 빌드 결과, 테스트와 Unity 파일을 Build Context에서 제외했습니다.
- 기존 PostgreSQL Compose에 API를 선택적으로 합치는 `compose.api.yaml`을 추가했습니다.
- 비루트 사용자, 읽기 전용 루트, capability 제거와 `no-new-privileges`를 적용했습니다.
- Migration을 별도 적용한 뒤 Production API 컨테이너의 `/health`와 게스트 생성을 검증했습니다.
- 전체 실행 순서를 [API 컨테이너 빌드와 실행](CONTAINER_DEPLOYMENT.md)에 문서화했습니다.

#### 완료 기준

- Docker 이미지가 재현 가능하게 빌드됩니다.
- 컨테이너에서 API가 실행됩니다.
- `/health`로 상태 확인이 가능합니다.

### Hardening Step 5. DB Readiness Health Check

상태: 완료

#### 왜 필요한가?

현재 `/health`는 API 프로세스 생존 여부 중심입니다. 운영에서는 DB 연결 가능 여부도 확인해야 배포와 모니터링이 쉬워집니다.

#### 할 일

- PostgreSQL 연결 상태를 확인하는 readiness endpoint 추가
- `/health`와 `/ready` 역할 분리 검토
- DB 장애 시 응답 상태 테스트

구현 결과:

- `/health`는 외부 의존성을 제외한 API liveness로 유지했습니다.
- `/ready`는 EF Core로 PostgreSQL 실제 연결을 확인하고 실패 시 503을 반환합니다.
- readiness 검사 시간을 최대 3초로 제한했습니다.
- Docker `HEALTHCHECK`가 `/ready`를 사용하도록 연결했습니다.
- 실제 컨테이너에서 DB 중지와 복구 시 200·503·200 전환을 검증했습니다.
- 배포 플랫폼 사용 기준을 [Liveness와 Readiness Health Check](HEALTH_CHECKS.md)에 기록했습니다.

#### 완료 기준

- API 프로세스 생존과 DB 준비 상태를 구분할 수 있습니다.
- 배포 환경에서 readiness check로 사용할 수 있습니다.

### Hardening Step 6. 장비 또는 스킬 시스템

#### 왜 필요한가?

게임 콘텐츠 관점의 첫 확장입니다. 장비/스킬은 서버 권위형 능력치 계산과 인벤토리/장착 검증을 보여 주기 좋습니다.

#### 할 일

- 장비 또는 스킬 중 하나를 먼저 선택
- 마스터 데이터와 플레이어 보유 데이터 분리
- 장착/강화/해제 API 설계
- 영웅 전투력 계산에 반영
- 스테이지 도전과 연결

#### 완료 기준

- 장비 또는 스킬 효과가 서버 전투력 계산에 반영됩니다.
- 클라이언트가 조작한 능력치를 서버가 신뢰하지 않습니다.
- Domain/Application/API/Infrastructure 테스트가 추가됩니다.

### Hardening Step 7. Redis 도입 후보 검토

#### 왜 바로 도입하지 않나?

Redis는 강력하지만 캐시 정합성, 장애 처리, TTL 정책이 추가됩니다. 먼저 어떤 기능에 필요한지 정한 뒤 도입하는 것이 좋습니다.

#### 도입 후보

- Rate Limit 카운터
- 랭킹
- 이벤트 참여 상태
- 짧은 TTL의 임시 중복 방지
- 자주 읽는 마스터 데이터 캐시

#### 완료 기준

- Redis가 필요한 기능과 필요하지 않은 기능을 구분합니다.
- Redis 장애 시 동작 기준을 문서화합니다.
- 재화 원장은 계속 PostgreSQL을 기준으로 유지합니다.

### Hardening Step 8. 모의 상점과 구매 이력

#### 왜 모의 결제부터 하나?

실제 결제는 플랫폼 영수증 검증, 환불, 보안, 법적 책임이 큽니다. 포트폴리오에서는 실제 돈이 오가지 않는 모의 상점으로 구매 이력과 중복 지급 방어를 먼저 보여 주는 것이 적합합니다.

#### 할 일

- 상품 카탈로그 정의
- 모의 구매 API
- 구매 영수증/구매 이력 저장
- 중복 구매 지급 방지
- 재화 원장과 연결

#### 완료 기준

- 같은 구매 요청이 중복 지급되지 않습니다.
- 구매 이력과 재화 지급 이력이 함께 남습니다.
- 실제 결제 연동과 모의 결제의 차이를 문서화합니다.

### Hardening Step 9. 관측성 고도화

#### 왜 필요한가?

현재는 기본 요청 로깅과 전역 예외 처리만 있습니다. 운영에서는 이상 징후와 성능 문제를 빠르게 파악해야 합니다.

#### 할 일

- 구조화 로그 필드 정리
- API별 응답 시간/오류율 메트릭 계획
- 보상 지급량 이상 탐지 후보 정리
- TraceId를 클라이언트/서버 로그에서 연결하는 방식 정리

#### 완료 기준

- 어떤 지표를 봐야 운영 문제를 발견할 수 있는지 문서화됩니다.
- 장애나 악용 의심 상황에서 확인할 로그 기준이 정리됩니다.

## 4. Unity 클라이언트와의 관계

서버 고도화와 Unity 클라이언트는 둘 다 다음 단계가 될 수 있습니다. 다만 포트폴리오 관점에서는 Unity 클라이언트를 먼저 붙이는 것이 시각적인 완성도가 높습니다.

추천 순서는 다음 중 하나입니다.

```text
선택 A: Unity 클라이언트 먼저
서버 MVP → Unity 클라이언트 → 장비/스킬 → 서버 고도화
```

```text
선택 B: 서버 고도화 먼저
서버 MVP → 재화 원장 → Rate Limiting → 관리자 조회 API → Unity 클라이언트
```

현재 사용자가 서버 고도화를 원한다면 선택 B로 진행합니다.

## 5. 다음 Step 추천

다음 서버 고도화 Step으로는 **장비 시스템**을 추천합니다.

이유:

- 운영 기반 Step 1~5가 완료되어 이제 게임 콘텐츠 확장을 안전하게 시작할 수 있습니다.
- 장비는 마스터 데이터와 플레이어 보유 데이터를 분리하는 구조를 학습하기 좋습니다.
- 획득·장착·해제와 서버 권위 전투력 계산을 통해 클라이언트 조작 방어를 보여 줄 수 있습니다.
- 이후 스킬, 상점과 드롭 보상으로 자연스럽게 확장할 수 있습니다.

다음 작업명:

```text
Hardening Step 8. 모의 상점과 구매 이력
```

Hardening Step 6에서는 장비 마스터와 플레이어 보유 데이터 분리, 장착·해제 API, 멱등 영수증, 슬롯 유일성 제약, 전투력 반영과 계층별 테스트를 완료했습니다. 상세 내용은 [장비 시스템](EQUIPMENT_SYSTEM.md)을 참고합니다.

Hardening Step 7에서는 현재 단일 인스턴스에 Redis를 추가하지 않기로 결정했습니다. PostgreSQL은 재화·멱등·영속 상태의 원본으로 유지하고, 다중 인스턴스 Rate Limit이 필요해질 때 Redis 또는 API Gateway를 우선 검토합니다. 기능별 장애 정책과 재검토 조건은 [Redis 도입 의사결정](REDIS_DECISION.md)에 정리했습니다.
