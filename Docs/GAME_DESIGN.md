# 게임 설계

## 1. 콘셉트

프로젝트 이름은 임시로 **Idle Guild**를 사용합니다.

플레이어는 모험가 길드의 관리자가 되어 영웅을 성장시키고 더 높은 스테이지를 공략합니다. 접속하지 않은 동안에도 골드가 쌓이며, 다음 접속 시 서버가 경과 시간을 검증해 보상을 지급합니다.

## 2. 핵심 플레이 루프

1. 영웅이 시간에 따라 골드를 생산한다.
2. 플레이어가 누적된 방치 보상을 수령한다.
3. 골드를 사용해 영웅의 레벨을 올린다.
4. 성장한 영웅으로 스테이지를 공략한다.
5. 더 높은 스테이지가 열리며 생산 효율이 증가한다.

이 흐름을 반복하면서 성장 속도와 도달 스테이지를 높이는 것이 게임의 기본 목표입니다.

## 3. MVP 기능

### 계정

- 별도 회원가입 없이 게스트 계정을 생성한다.
- 발급된 토큰으로 자신의 게임 데이터만 조회하고 변경한다.

### 게임 상태

- 현재 골드, 영웅 레벨, 최고 스테이지를 조회한다.
- 시간 계산은 클라이언트 시간이 아닌 서버의 UTC 시간을 사용한다.

### 방치 보상

- 서버가 마지막 정산 시각부터 현재까지의 경과 시간을 계산한다.
- 최대 누적 시간은 초기값 8시간으로 제한한다.
- 보상은 `경과한 전체 초 × 초당 골드 생산량`으로 계산한다.
- 보상 지급과 마지막 정산 시각 변경은 하나의 트랜잭션으로 처리한다.
- 같은 요청이 재전송돼도 보상이 중복 지급되지 않아야 한다.
- 클라이언트는 수령 요청마다 최대 64자의 `Idempotency-Key`를 보내며 재시도할 때 같은 값을 사용한다.

### 영웅 성장

- 플레이어는 골드를 소비해 영웅 레벨을 1씩 올린다.
- 강화 비용과 생산량은 서버의 규칙으로 계산한다.
- 골드가 부족하면 상태를 변경하지 않고 명확한 오류를 반환한다.

### 스테이지

- 서버가 영웅의 전투력과 스테이지 요구 전투력을 비교한다.
- 성공하면 최고 스테이지와 생산 보너스를 갱신한다.
- MVP에서는 실시간 전투 루프 대신 결정론적 결과 계산을 사용한다.

## 4. 초기 규칙

초기 수치는 구조 검증을 위한 값이며 밸런스 테스트 후 변경할 수 있습니다.

| 항목 | 초기 규칙 |
| --- | --- |
| 기본 초당 골드 | `1` |
| 최대 오프라인 누적 | `8시간` |
| 영웅 전투력 | `10 × 영웅 레벨` |
| 강화 비용 | `floor(10 × 1.15^(현재 레벨 - 1))` |
| 최대 영웅 레벨 | `297` |
| 스테이지 요구 전투력 | `floor(10 × 1.2^(스테이지 - 1))` |
| 스테이지 생산 보너스 | 최고 스테이지당 `5%` |

재화 계산에는 부동소수점 오차가 개입하지 않도록 정수 연산 또는 명시적인 반올림 규칙을 사용합니다.

강화 요청은 최대 64자의 `Idempotency-Key`를 사용합니다. 성공뿐 아니라 골드 부족과 최대 레벨 판정도 저장해 같은 키의 재전송 결과가 바뀌지 않게 합니다.

## 5. 서버 권위 원칙

클라이언트는 행동을 요청하고 결과를 표현하는 역할을 맡습니다. 다음 값은 서버가 직접 계산하거나 검증합니다.

- 현재 시각과 오프라인 경과 시간
- 지급되는 골드
- 강화 비용과 차감 결과
- 전투력과 스테이지 성공 여부
- 최종 저장 상태

## 6. MVP에서 제외하는 기능

- 소셜 로그인과 계정 연동
- 장비, 스킬, 영웅 수집
- 길드와 채팅
- PvP 및 랭킹
- 결제와 광고
- Redis, 메시지 큐, 마이크로서비스

제외 기능은 MVP가 안정적으로 완성된 뒤 학습 가치에 따라 하나씩 확장합니다.
