# 스테이지 진행

## 1. 구현 목표

실시간 전투 대신 영웅 전투력과 스테이지 요구 전투력을 비교하는 결정론적 판정을 사용합니다. 같은 상태와 요청은 언제 실행해도 같은 전투 결과를 만들며, 성공한 스테이지는 방치 골드 생산량을 높입니다.

## 2. 전투력 규칙

```text
영웅 전투력 = 10 × 영웅 레벨
스테이지 요구 전투력 = floor(10 × 1.2^(스테이지 - 1))
```

| 스테이지 | 요구 전투력 |
| ---: | ---: |
| 1 | 10 |
| 2 | 12 |
| 3 | 14 |
| 4 | 17 |
| 32 | 2,848 |

요구 전투력은 `10 × 120^(stage-1) ÷ 100^(stage-1)`을 `BigInteger`로 계산해 부동소수점 차이를 제거합니다. 최대 영웅 레벨 297의 전투력은 2,970이며 현재 규칙으로 클리어 가능한 마지막 스테이지는 32입니다.

## 3. 진행 규칙

- 새 플레이어는 스테이지 1을 완료한 상태에서 시작합니다.
- 현재 최고 스테이지의 바로 다음 스테이지만 새로 완료할 수 있습니다.
- 이전 스테이지 도전은 `alreadyCompleted`입니다.
- 한 단계 이상 건너뛴 도전은 `stageLocked`입니다.
- 다음 스테이지에서 전투력이 부족하면 `insufficientPower`입니다.
- 전투력이 요구량 이상이면 `succeeded`이며 최고 스테이지가 증가합니다.

실패도 최초 판정을 영수증으로 저장하므로 같은 키의 재요청 결과는 나중에 영웅이 성장해도 바뀌지 않습니다.

## 4. API 사용

```http
POST /api/v1/stages/2/challenge
Authorization: Bearer {access-token}
Idempotency-Key: stage-2-20260707-001
```

성공은 `200 OK`, 게임 규칙상 실패는 `409 Conflict`입니다.

```json
{
  "idempotencyKey": "stage-2-20260707-001",
  "targetStage": 2,
  "outcome": "succeeded",
  "previousHighestStage": 1,
  "highestStageAfter": 2,
  "heroPower": 20,
  "requiredPower": 12,
  "productionBonusPercentAfter": 5,
  "checkpointGoldAwarded": 100,
  "goldBalanceAfter": 190,
  "processedAtUtc": "2026-07-07T00:03:20+00:00",
  "isReplay": false
}
```

하나의 멱등 키를 다른 목표 스테이지에 재사용하면 `409 Problem Details`를 반환합니다. 저장 충돌이 세 번의 내부 재시도 뒤에도 계속되면 `503 Service Unavailable`을 반환합니다.

## 5. 생산 보너스

스테이지 1은 기준 생산량 100%이며, 이후 완료한 스테이지마다 5%를 더합니다.

```text
생산 보너스 = (최고 스테이지 - 1) × 5%
실제 생산 배율 = 100% + 생산 보너스
```

스테이지 2에서는 100초 동안 `floor(100 × 1골드 × 105 / 100) = 105골드`를 받습니다.

### 소수 골드 이월

1초 동안 105% 생산을 계산하면 1.05골드입니다. 매번 0.05를 버리면 짧게 자주 수령하는 플레이어가 손해를 보므로 `idle_reward_remainder_hundredths`에 1/100 골드 단위 잔여값을 저장합니다.

스테이지 2에서 1초씩 20번 수령해도 잔여값이 누적되어 총 21골드를 받습니다.

### 보너스 소급 적용 방지

스테이지 성공 전에 쌓인 방치 시간을 새 보너스로 계산하면 다음 순서로 과거 보상을 부풀릴 수 있습니다.

1. 스테이지 1에서 장시간 방치합니다.
2. 보상을 받기 전에 스테이지 2를 완료합니다.
3. 과거 전체 시간에 5% 보너스를 적용합니다.

이를 막기 위해 성공 순간까지의 방치 보상을 기존 배율로 먼저 체크포인트 정산하고, 그 시각 이후부터 새 배율을 적용합니다. 응답의 `checkpointGoldAwarded`가 이 정산량입니다.

## 6. 동시 요청 처리

같은 다음 스테이지에 서로 다른 키 두 개가 동시에 도착하면 둘 다 이전 상태를 읽을 수 있습니다. 먼저 저장한 요청이 `xmin`을 바꾸며, 나중 요청은 충돌 후 최신 상태를 다시 읽습니다.

- 첫 요청: `succeeded`
- 두 번째 요청: 재판정 후 `alreadyCompleted`
- 최종 최고 스테이지: 한 단계만 증가

같은 키의 동시 요청은 `(player_id, idempotency_key)` 복합 기본키가 영수증을 하나만 허용합니다.

## 7. 계층별 역할

- Domain: 전투력, 요구 전투력, 진행 순서, 생산 배율과 소수 이월 계산
- Application: 멱등 영수증, 경로 파라미터 키 충돌, 최대 3회 저장 재시도
- Infrastructure: 스테이지 영수증, 잔여 골드 열, 체크 제약과 `xmin`
- API: JWT, 스테이지 범위, 멱등 키 검증과 HTTP 결과 변환
- Tests: 결정론적 수치, 체크포인트, 소수 이월, 실제 동시 도전 검증
