# Step별 진행 이유와 의사결정

이 문서는 각 Step에서 왜 해당 작업을 진행했는지, 어떤 문제를 해결하려 했는지, 어떤 선택을 했고 어떻게 검증했는지 정리합니다.

`ROADMAP.md`가 “무엇을 할 것인가”를 보여 준다면, 이 문서는 “왜 그렇게 진행했는가”를 보여 주기 위한 문서입니다.

## Step 1. 프로젝트 설계

### 왜 진행했나?

처음부터 기능을 바로 구현하면 방치형 게임 서버에서 무엇을 서버가 책임져야 하는지 흐려질 수 있습니다. 이 프로젝트의 목적은 단순히 API를 만드는 것이 아니라 서버 권위형 게임 구조를 학습하고 포트폴리오로 설명할 수 있게 만드는 것입니다.

### 어떤 문제를 해결했나?

- 클라이언트와 서버의 책임 경계가 불명확한 문제
- MVP 범위가 계속 커지는 문제
- 이후 Step의 우선순위를 판단하기 어려운 문제

### 어떤 선택을 했나?

- 방치형 모험가 길드라는 작은 게임 콘셉트를 정했습니다.
- 핵심 루프를 `골드 생산 → 보상 수령 → 영웅 강화 → 스테이지 진행 → 생산 보너스`로 제한했습니다.
- 클라이언트가 아니라 서버가 시간, 재화, 성장 결과를 계산하는 원칙을 세웠습니다.
- Unity 클라이언트는 서버 MVP 이후 별도 Step으로 분리했습니다.

### 어떻게 검증했나?

- `Docs/GAME_DESIGN.md`에 게임 규칙과 MVP 범위를 작성했습니다.
- `Docs/ARCHITECTURE.md`에 서버 구조와 계층 책임을 작성했습니다.
- `Docs/ROADMAP.md`에 Step별 완료 기준을 정의했습니다.

## Step 2. 서버 기반 구성

### 왜 진행했나?

게임 기능을 구현하기 전에 실행 가능한 서버 골격과 테스트 구조가 필요했습니다. 기반이 없으면 이후 기능이 “돌아가는지”, “어디에 코드를 넣어야 하는지”, “어떻게 검증해야 하는지”가 매번 흔들립니다.

### 어떤 문제를 해결했나?

- API, Domain, Infrastructure 코드가 한 프로젝트에 섞이는 문제
- 테스트가 나중에 붙어 구조 변경 비용이 커지는 문제
- 서버가 정상 실행되는지 확인할 최소 기준이 없는 문제

### 어떤 선택을 했나?

- ASP.NET Core Minimal API를 사용했습니다.
- `Api`, `Application`, `Domain`, `Infrastructure`, `Tests`로 프로젝트를 분리했습니다.
- Health Check와 OpenAPI를 초기부터 넣었습니다.
- `Directory.Build.props`로 Nullable, 경고 오류 처리 같은 품질 기준을 공통 적용했습니다.

### 어떻게 검증했나?

- `/health`가 정상 응답하는지 테스트했습니다.
- OpenAPI 문서가 생성되는지 확인했습니다.
- 전체 솔루션 빌드와 테스트가 가능한 상태로 만들었습니다.

## Step 3. PostgreSQL 연동

### 왜 진행했나?

방치형 게임 서버는 플레이어의 골드, 레벨, 마지막 보상 정산 시각 같은 상태를 안전하게 저장해야 합니다. 메모리 저장만으로는 서버 재시작, 동시 요청, 데이터 무결성 문제를 다룰 수 없습니다.

### 어떤 문제를 해결했나?

- 플레이어 상태를 영속화할 수 없는 문제
- 잘못된 재화나 레벨 값이 저장될 수 있는 문제
- 실제 DB와 코드 모델이 어긋날 수 있는 문제

### 어떤 선택을 했나?

- PostgreSQL을 기본 DB로 선택했습니다.
- Docker Compose로 로컬 DB 실행 환경을 만들었습니다.
- EF Core DbContext와 Migration으로 스키마를 관리했습니다.
- `PlayerGameState`를 첫 저장 모델로 만들고 DB 제약조건을 추가했습니다.
- PostgreSQL의 `xmin`을 낙관적 동시성 토큰으로 사용했습니다.

### 어떻게 검증했나?

- 실제 PostgreSQL에 저장하고 다른 DbContext로 다시 읽는 통합 테스트를 작성했습니다.
- Migration이 빈 DB를 현재 스키마로 만들 수 있는지 확인했습니다.
- 음수 골드, 잘못된 레벨 같은 값이 저장되지 않도록 제약조건을 검토했습니다.

## Step 4. 게스트 인증

### 왜 진행했나?

게임 상태가 DB에 저장되기 시작하면 “누가 자신의 상태를 조회하고 변경하는가”를 구분해야 합니다. 클라이언트가 플레이어 ID를 마음대로 보내는 구조는 다른 사용자의 데이터에 접근할 위험이 있습니다.

### 어떤 문제를 해결했나?

- 사용자 식별 없이 모든 상태에 접근할 수 있는 문제
- 요청 Body나 URL의 플레이어 ID를 신뢰할 때 생기는 보안 문제
- Unity 클라이언트가 빠르게 시작할 로그인 흐름이 없는 문제

### 어떤 선택을 했나?

- 별도 회원가입 없이 게스트 계정을 생성하게 했습니다.
- JWT의 `sub` Claim에 플레이어 ID를 넣었습니다.
- 보호된 API는 요청에서 플레이어 ID를 받지 않고 검증된 JWT에서만 읽게 했습니다.
- Application은 토큰 구현을 모르도록 `IAccessTokenIssuer` 추상화를 두었습니다.

### 어떻게 검증했나?

- 토큰 없는 요청이 401을 반환하는지 테스트했습니다.
- 게스트 토큰으로 본인 상태만 조회되는지 테스트했습니다.
- 서로 다른 게스트 토큰이 서로 다른 상태를 읽는지 테스트했습니다.
- 변조된 JWT가 거부되는지 테스트했습니다.

## Step 5. 방치 보상

### 왜 진행했나?

방치형 게임의 핵심은 접속하지 않은 시간 동안 보상이 쌓이고, 다음 접속 시 그 보상을 받는 경험입니다. 이 기능이 없으면 프로젝트가 “방치형 게임 서버”라고 설명되기 어렵습니다.

### 어떤 문제를 해결했나?

- 클라이언트 시간을 믿으면 시간 조작으로 골드를 부정 획득할 수 있는 문제
- 네트워크 재시도로 같은 보상이 여러 번 지급될 수 있는 문제
- 동시에 보상 요청이 들어왔을 때 골드가 중복 지급될 수 있는 문제
- 무제한 오프라인 누적으로 재화가 과도하게 쌓이는 문제

### 어떤 선택을 했나?

- 서버 UTC 시간을 기준으로 경과 시간을 계산했습니다.
- 최대 누적 시간을 8시간으로 제한했습니다.
- 보상 수령 요청에 `Idempotency-Key`를 요구했습니다.
- 최초 지급 결과를 `IdleRewardClaimReceipt`로 저장해 같은 키 재시도는 같은 결과를 반환하게 했습니다.
- 저장 충돌은 Application에서 재시도하고, DB 충돌은 Infrastructure에서 Application 예외로 변환했습니다.

### 어떻게 검증했나?

- 순수 계산 규칙을 Domain 테스트로 검증했습니다.
- Handler의 멱등 재생을 Application 테스트로 검증했습니다.
- HTTP 계약을 API 테스트로 검증했습니다.
- 실제 PostgreSQL에서 동시 요청이 한 번만 지급되는지 통합 테스트로 검증했습니다.

## Step 6. 영웅 성장

### 왜 진행했나?

보상을 받은 뒤 사용할 곳이 있어야 게임 루프가 이어집니다. 영웅 성장은 방치 보상으로 얻은 골드를 소비하게 만들고, 이후 스테이지 진행의 기반 능력치를 제공합니다.

### 어떤 문제를 해결했나?

- 골드를 획득해도 성장 목표가 없는 문제
- 강화 비용 계산과 골드 차감이 클라이언트 조작에 취약할 수 있는 문제
- 성공한 강화뿐 아니라 실패한 강화 요청도 재시도 때 결과가 바뀔 수 있는 문제
- 동시 강화 요청으로 한 번의 골드가 여러 번 쓰일 수 있는 문제

### 어떤 선택을 했나?

- 강화 비용을 서버에서 결정론적으로 계산했습니다.
- 골드 부족, 최대 레벨, 성공을 명확한 결과로 구분했습니다.
- 성공뿐 아니라 실패 결과도 영수증으로 저장했습니다.
- 보상 API와 같은 멱등 키 패턴을 재사용했습니다.
- 비용 계산은 부동소수점 오차를 피하도록 정수 기반 규칙으로 관리했습니다.

### 어떻게 검증했나?

- 강화 비용과 최대 레벨 규칙을 Domain 테스트로 검증했습니다.
- 같은 키 재시도와 실패 재생을 Application/API 테스트로 검증했습니다.
- 실제 PostgreSQL 동시 강화 요청에서 골드가 중복 차감되지 않는지 검증했습니다.

## Step 7. 스테이지 진행

### 왜 진행했나?

영웅 성장만 있으면 골드를 소비하는 구조는 생기지만, 성장의 목표가 약합니다. 스테이지 진행은 “강해져서 더 높은 단계를 여는” 목표를 만들고, 생산 보너스로 다시 방치 보상에 영향을 주는 완성된 루프를 만듭니다.

### 어떤 문제를 해결했나?

- 성장의 목적지가 부족한 문제
- 전투 결과를 클라이언트가 조작할 수 있는 문제
- 이미 완료한 스테이지나 잠긴 스테이지 요청을 어떻게 처리할지 불명확한 문제
- 스테이지 보너스를 과거 방치 시간에 소급 적용하면 보상이 부풀려지는 문제

### 어떤 선택을 했나?

- MVP에서는 실시간 전투 대신 결정론적 전투력 비교를 사용했습니다.
- 영웅 전투력과 스테이지 요구 전투력을 서버가 계산했습니다.
- 새로 완료할 수 있는 스테이지는 현재 최고 스테이지의 바로 다음 단계로 제한했습니다.
- 성공, 전투력 부족, 이미 완료, 잠김을 명확한 결과로 구분했습니다.
- 성공 순간 이전의 방치 보상은 기존 배율로 먼저 정산하고, 이후부터 새 생산 보너스를 적용했습니다.
- 1/100 골드 단위의 잔여값을 서버 상태에 이월해 짧은 보상 수령에서도 손실이 쌓이지 않게 했습니다.

### 어떻게 검증했나?

- 전투력과 요구 전투력 계산을 Domain 테스트로 검증했습니다.
- 스테이지 잠금, 실패, 성공, 멱등 재생을 Application/API 테스트로 검증했습니다.
- 실제 PostgreSQL에서 동시 스테이지 도전이 한 번만 진행되는지 검증했습니다.
- 생산 보너스와 잔여값이 DB에 보존되는지 검증했습니다.

## Step 8-1. API 오류 계약 정리

### 왜 진행했나?

서버 기능이 늘어나면 오류 응답도 늘어납니다. Endpoint마다 다른 형식의 오류를 반환하면 Unity 클라이언트가 예외 상황을 처리하기 어렵고, 포트폴리오를 보는 사람도 API 계약을 신뢰하기 어렵습니다.

### 어떤 문제를 해결했나?

- `BadRequest(string)`, `Conflict(string)`, `ProblemDetails`가 섞여 있던 문제
- 멱등 키 검증 코드가 여러 Endpoint에 반복되던 문제
- 게임 규칙 실패와 요청 계약 위반이 구분되지 않는 문제

### 어떤 선택을 했나?

- 일반 오류는 `ProblemDetails` 중심으로 통일했습니다.
- 멱등 키 검증과 공통 오류 생성을 `EndpointProblemResults`로 모았습니다.
- 게임 규칙상 정상 판정된 실패는 기능 응답 DTO의 `outcome`으로 유지했습니다.
- 요청 자체가 잘못된 경우는 `ProblemDetails`로 반환했습니다.

### 어떻게 검증했나?

- API 테스트에서 상태 코드뿐 아니라 `ProblemDetails.title`까지 확인했습니다.
- `Docs/API_ERRORS.md`에 클라이언트 처리 기준을 문서화했습니다.

## Step 8-2. 로깅과 전역 예외 처리

### 왜 진행했나?

포트폴리오 프로젝트라도 서버답게 운영 중 문제를 추적할 수 있어야 합니다. 예상한 오류는 Endpoint가 처리하지만, 예상하지 못한 예외가 그대로 노출되면 보안과 사용자 경험 모두 좋지 않습니다.

### 어떤 문제를 해결했나?

- 처리되지 않은 예외가 클라이언트에 내부 정보를 노출할 수 있는 문제
- 서버 로그와 클라이언트 오류를 연결할 추적 정보가 없는 문제
- 요청 흐름을 확인할 최소 로그가 없는 문제
- Authorization 헤더나 Body를 무심코 로그에 남길 위험

### 어떤 선택을 했나?

- `GlobalExceptionHandler`를 추가해 예상하지 못한 예외를 500 `ProblemDetails`로 변환했습니다.
- 응답에 `traceId`를 포함해 클라이언트 오류와 서버 로그를 연결할 수 있게 했습니다.
- Development 환경에서는 학습 편의를 위해 예외 메시지를 보여 주고, 운영 환경에서는 안전한 안내만 반환하게 했습니다.
- `HttpLogging`은 Method, Path, StatusCode, Duration만 기록하게 제한했습니다.

### 어떻게 검증했나?

- 테스트용 Repository가 강제로 예외를 던지게 만들어 실제 HTTP 파이프라인에서 500 `ProblemDetails`가 반환되는지 검증했습니다.
- `Docs/OBSERVABILITY.md`에 로깅과 예외 처리 원칙을 문서화했습니다.

## Step 8-3. 데모 시나리오와 배포 설정 문서화

### 왜 진행했나?

서버 기능과 운영 기본기는 갖춰졌지만, 제3자가 README만 보고 핵심 시나리오를 직접 실행해 보는 흐름은 더 보강할 필요가 있었습니다. 포트폴리오는 코드뿐 아니라 “어떻게 실행하고 무엇을 확인하면 되는지”가 분명해야 합니다.

### 어떤 문제를 해결했나?

- 데모 시나리오가 README에 충분히 친절하게 정리되어 있지 않은 문제
- 배포 환경에서 어떤 설정값이 필요한지 한눈에 보기 어려운 문제
- 운영 환경에서 Development와 어떤 동작이 달라지는지 설명이 부족한 문제
- 수동 API 호출 파일이 멱등 재시도와 오류 계약 예시까지 보여 주지 못한 문제

### 어떤 선택을 했나?

- README에는 빠른 데모 흐름만 두고, 자세한 명령은 `Docs/DEMO_SCENARIO.md`로 분리했습니다.
- 운영 환경 변수, Migration, Health Check, 로그 기준은 `Docs/DEPLOYMENT.md`로 분리했습니다.
- `IdleGuild.Api.http`에 멱등 재시도와 오류 계약 확인 요청을 추가했습니다.
- 아직 특정 클라우드나 컨테이너 빌드 파일에 묶지 않고, 배포 대상이 정해져도 재사용 가능한 공통 운영 기준을 먼저 문서화했습니다.

### 어떻게 검증했나?

- README에서 데모 문서와 HTTP Client 파일로 이동할 수 있게 링크를 추가했습니다.
- 데모 문서가 게스트 생성부터 생산 보너스 확인까지 핵심 루프를 빠짐없이 설명하는지 검토했습니다.
- 배포 문서가 운영 필수 환경 변수와 체크리스트를 포함하는지 검토했습니다.

## Step 8-4. 서버 MVP 완료 선언

### 왜 진행했나?

서버 기능, 오류 계약, 운영 기본기, 데모 문서가 모두 갖춰졌으므로 서버 파트를 계속 열린 상태로 두기보다 MVP 완료 기준을 명확히 표시할 필요가 있었습니다. 그래야 다음 세션에서 Unity 클라이언트로 자연스럽게 넘어갈 수 있습니다.

### 어떤 문제를 해결했나?

- 서버 MVP 완료 선언과 이후 Unity 클라이언트 전환 기준을 명확히 표시해야 하는 문제
- 현재 포함된 기능과 의도적으로 제외한 범위가 한곳에 정리되어 있지 않은 문제
- 서버 테스트 범위와 완료 근거를 빠르게 확인하기 어려운 문제
- 이후 확장 후보와 현재 MVP 경계가 섞여 보일 수 있는 문제

### 어떤 선택을 했나?

- `Docs/SERVER_MVP_COMPLETE.md`를 추가해 완료 기준, 포함 기능, 제외 범위, 테스트 기준, 다음 확장 후보를 정리했습니다.
- README 현재 상태를 서버 MVP 완료로 갱신했습니다.
- Roadmap의 Step 8을 완료 상태로 바꾸고 Step 9를 대기 중으로 표시했습니다.
- 제외 범위는 결함이 아니라 의도적인 MVP 경계로 설명했습니다.

### 어떻게 검증했나?

- 전체 테스트로 서버 기능이 여전히 통과하는지 확인했습니다.
- README, ROADMAP, STEP_DECISIONS, PROJECT_STRUCTURE의 상태 표현이 같은 방향을 가리키는지 확인했습니다.
- Unity 클라이언트가 참고할 서버 사용 기준을 문서화했습니다.

## Hardening Step 1. 재화 변경 이력 원장

### 왜 진행했나?

현재 골드 잔액만 저장하면 운영 중 골드가 어떤 기능에서 얼마나 증가하거나 감소했는지 설명할 수 없습니다. 재화 버그, 중복 지급 의심, 유저 문의에 대응하려면 최종 상태와 별도로 변경 과정을 보존해야 합니다.

### 어떤 문제를 해결했나?

- 방치 보상, 영웅 강화, 스테이지 체크포인트의 골드 변경 원인을 추적할 수 없는 문제
- 게임 상태는 저장됐지만 감사 이력이 누락될 수 있는 문제
- 같은 멱등 요청이나 동시 요청이 원장을 중복 생성할 수 있는 문제
- 변경 전·후 잔액과 증감량이 서로 맞지 않는 잘못된 감사 데이터가 저장될 수 있는 문제

### 어떤 선택을 했나?

- `GoldLedgerEntry`를 Domain 객체로 만들고 변경 전 잔액, 증감량, 변경 후 잔액의 등식을 검증했습니다.
- 실제 골드가 변한 요청만 원장에 기록하고 0원 요청과 실패 판정은 기존 기능 영수증으로 추적했습니다.
- 원장 참조 ID로 정규화된 `Idempotency-Key`를 사용했습니다.
- `(player_id, reason, reference_id)` 유일 인덱스로 기능별 중복 원장을 방지했습니다.
- 게임 상태, 기능 영수증, 원장을 같은 `SaveChangesAsync`에서 원자적으로 저장했습니다.
- 일반 클라이언트에는 원장을 노출하지 않고 이후 관리자 조회 API에서 읽도록 경계를 정했습니다.

### 어떻게 검증했나?

- Domain 테스트에서 정상 증가, 잘못된 잔액 등식, 0원 변경 거부를 검증했습니다.
- Application 테스트에서 보상 증가, 강화 차감, 스테이지 체크포인트 원장의 전후 잔액을 검증했습니다.
- 실제 PostgreSQL 동시성 테스트에서 같은 보상 요청은 원장 하나만, 서로 다른 동시 강화는 성공한 차감 원장 하나만 남는지 검증했습니다.
- 전체 66개 테스트가 통과하는지 확인했습니다.

## Hardening Step 2. Rate Limiting

### 왜 진행했나?

멱등성은 중복 지급을 막지만 요청 처리 비용과 DB 접근 자체를 막지는 않습니다. 인증 없는 게스트 생성과 상태 변경 API를 자동화 도구가 과도하게 호출하면 정상 플레이와 저장소에 부담을 줄 수 있습니다.

### 어떤 문제를 해결했나?

- 같은 IP에서 게스트 계정을 무제한 생성할 수 있는 문제
- 한 플레이어가 보상·강화·스테이지 API를 과도하게 호출할 수 있는 문제
- 제한 초과 후 클라이언트가 언제 재시도해야 하는지 알 수 없는 문제
- IP만으로 상태 변경을 제한하면 같은 네트워크의 정상 플레이어가 서로 영향을 주는 문제

### 어떤 선택을 했나?

- ASP.NET Core 기본 Rate Limiting을 사용해 별도 외부 저장소 없이 단일 서버 방어를 먼저 구현했습니다.
- 게스트 생성은 연결 IP당 5회/분, 상태 변경은 검증된 JWT 플레이어당 30회/분으로 구분했습니다.
- 조회 API는 제한에서 제외해 화면 갱신이 게임 행동 한도를 소모하지 않게 했습니다.
- 대기열 없이 초과 요청을 즉시 429로 반환했습니다.
- 429 Body는 공통 `IProblemDetailsService`를 사용하고 `Retry-After`, `traceId`, `retryAfterSeconds`를 제공했습니다.
- 다중 인스턴스에서는 API Gateway 또는 Redis 공유 카운터가 필요하다는 경계를 문서화했습니다.

### 어떻게 검증했나?

- 같은 IP의 게스트 생성이 5회 허용된 뒤 429가 되는지 API 테스트로 검증했습니다.
- 429 Content-Type, 제목, 상태, 재시도 헤더와 확장값을 확인했습니다.
- 한 플레이어가 30회 한도를 소진한 뒤 같은 IP의 다른 플레이어는 정상 요청할 수 있는지 검증했습니다.
- 기존 인증·게임 API 테스트가 Rate Limiting 추가 후에도 통과하는지 확인했습니다.
- 실제 PostgreSQL 통합 테스트를 포함한 전체 68개 테스트가 통과하는지 확인했습니다.

## Hardening Step 3. 관리자용 조회 API

### 왜 진행했나?

골드 원장은 저장되고 있지만 운영자가 특정 플레이어의 현재 상태와 재화 변경 원인을 확인할 안전한 통로가 없었습니다. 유저 문의와 이상 징후 대응을 위해 상태를 변경하지 않는 관리자 조회 경계가 필요했습니다.

### 어떤 문제를 해결했나?

- 특정 플레이어의 현재 서버 상태를 운영 중 확인할 수 없는 문제
- 골드 원장을 DB에 직접 접속하지 않고 조회할 API가 없는 문제
- 게스트 JWT와 관리자 권한 경계가 구분되지 않은 문제
- 원장 전체를 한 번에 반환하면 메모리와 DB 부하가 커지는 문제
- 페이지 번호 조회 중 새 데이터가 추가되면 중복·누락될 수 있는 문제

### 어떤 선택을 했나?

- 기존 JWT 검증 뒤 `account_type=admin`을 요구하는 Authorization Policy를 추가했습니다.
- 공개 관리자 로그인이나 토큰 발급 Endpoint는 만들지 않고 외부 운영자 인증 시스템의 책임으로 남겼습니다.
- 플레이어 현재 상태와 골드 원장을 서로 다른 읽기 전용 Endpoint로 분리했습니다.
- 원장은 기본 50개, 최대 100개와 불투명한 다음 커서를 사용하는 키셋 페이지네이션으로 구현했습니다.
- `(occurred_at_utc, entry_id)`로 동일 시각의 순서도 안정적으로 결정했습니다.
- PostgreSQL 인덱스를 `(player_id, occurred_at_utc, entry_id)`로 확장했습니다.
- 관리자 조회는 관리자 `sub` 기준 120회/분으로 제한했습니다.

### 어떻게 검증했나?

- Application 테스트에서 관리자 상태 필드와 두 페이지의 중복 없는 연결을 검증했습니다.
- API 테스트에서 토큰 없음 401, 게스트 토큰 403, 관리자 조회 성공, 잘못된 커서 400을 검증했습니다.
- 실제 PostgreSQL에서 플레이어 격리, 최신순 정렬과 커서 조건의 SQL 번역을 검증했습니다.
- 새 인덱스 Migration을 적용한 전체 76개 테스트가 통과하는지 확인했습니다.
