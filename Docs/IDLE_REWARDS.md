# 방치 보상

## 1. 구현 목표

클라이언트가 알려 주는 시간이나 지급량을 신뢰하지 않고 서버가 UTC 시각과 저장 상태만으로 골드를 계산합니다. 네트워크 오류로 같은 요청이 반복되거나 두 요청이 동시에 도착해도 한 번만 지급해야 합니다.

## 2. 정산 규칙

- 마지막 정산 시각부터 서버 현재 시각까지의 전체 초를 계산합니다.
- 기본 생산량은 초당 1골드이며 스테이지 1 이후 단계당 5%가 증가합니다.
- 한 번에 반영하는 시간은 최대 8시간, 즉 28,800초입니다.
- 서버 시각이 이전 정산 시각보다 과거이면 0초로 처리합니다.
- 정산 후 골드와 마지막 정산 시각을 함께 변경합니다.
- 1골드 미만 결과는 1/100 골드 단위 잔여값으로 다음 정산에 이월합니다.

예를 들어 마지막 정산 후 2시간이 지났다면 `7,200초 × 1골드 = 7,200골드`를 지급합니다. 10시간이 지났다면 상한 때문에 28,800골드만 지급합니다.

## 3. API 사용

```http
POST /api/v1/rewards/idle/claim
Authorization: Bearer {access-token}
Idempotency-Key: claim-20260705-001
```

요청 Body는 없습니다. `Idempotency-Key`는 비어 있지 않은 최대 64자 문자열이어야 합니다. 하나의 사용자 행동을 재시도할 때는 반드시 같은 키를 보내고, 새로운 수령 행동에는 새로운 키를 사용합니다.

```json
{
  "idempotencyKey": "claim-20260705-001",
  "goldAwarded": 7200,
  "accumulatedSeconds": 7200,
  "goldBalanceAfter": 7200,
  "remainderHundredths": 0,
  "productionPercent": 100,
  "claimedAtUtc": "2026-07-05T02:00:00+00:00",
  "isReplay": false
}
```

같은 키를 다시 보내면 최초 응답과 같은 지급량·잔액·정산 시각을 반환하며 `isReplay`만 `true`가 됩니다.

저장 충돌이 짧은 내부 재시도 후에도 계속되면 API는 `409 Conflict`를 반환합니다. 클라이언트는 잠시 후 같은 `Idempotency-Key`로 다시 요청합니다.

## 4. 중복과 동시 요청 방어

```text
요청
  -> 기존 영수증 조회
     -> 있음: 최초 결과 반환
     -> 없음: 최신 상태로 보상 계산
              -> 상태 변경 + 영수증 추가
              -> 하나의 SaveChanges로 저장
```

두 요청이 동시에 계산하더라도 `player_game_states.xmin`이 상태 충돌을 감지하고, 영수증 복합 기본키가 같은 멱등 키의 중복 저장을 차단합니다. 충돌한 요청은 EF 변경 추적을 비운 뒤 최신 상태와 영수증을 다시 읽습니다.

## 5. 계층별 역할

- Domain: 경과 시간, 8시간 상한, 지급 골드 계산
- Application: 영수증 확인, 작업 순서, 최대 3회 충돌 재시도
- Infrastructure: EF 저장소, `xmin`, 복합 기본키, PostgreSQL 오류 변환
- API: JWT 사용자 확인, `Idempotency-Key` 검증, HTTP 응답 변환
- Tests: 규칙 경계값, 같은 키 재요청, 실제 DB 동시 요청 검증
