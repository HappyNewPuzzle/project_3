# 모의 상점과 구매 이력

## 1. 목적

실제 돈을 받지 않고도 상품 카탈로그, 구매 승인, 중복 지급 방지, 구매 영수증과 재화 원장을 학습하기 위한 기능입니다. 상품 가격의 `mockPrice`는 테스트용 숫자이며 결제 수단으로 청구되지 않습니다.

## 2. 상품

| 상품 ID | 이름 | 모의 가격 | 지급 골드 |
| --- | --- | ---: | ---: |
| `small-gold-pack` | Small Gold Pack | 100 | 100 |
| `large-gold-pack` | Large Gold Pack | 450 | 500 |

클라이언트는 상품 ID만 선택합니다. 가격과 지급량은 서버 `ShopCatalog`에서 결정하므로 요청 조작으로 골드를 늘릴 수 없습니다.

## 3. API

- `GET /api/v1/shop/products`: 판매 상품 조회
- `POST /api/v1/shop/products/{productId}/purchase`: 모의 구매와 골드 지급
- `GET /api/v1/shop/purchases`: 인증된 플레이어의 최신순 구매 이력

구매 요청에는 최대 64자의 `Idempotency-Key`가 필요합니다. 같은 키와 상품을 재전송하면 최초 구매가 재생되고 골드는 다시 지급되지 않습니다. 같은 키를 다른 상품에 사용하면 409를 반환합니다.

## 4. 원자성

한 번의 PostgreSQL 트랜잭션에서 다음 세 변경을 함께 저장합니다.

1. `player_game_states.gold` 증가
2. `shop_purchase_receipts` 구매 영수증 추가
3. `gold_ledger_entries`에 `ShopPurchase` 지급 원장 추가

일부만 저장되는 상태를 허용하지 않습니다. 동시 요청 충돌은 최신 상태를 다시 읽어 제한 횟수만 재시도합니다.

## 5. 실제 결제와의 차이

현재 구현은 포트폴리오용 모의 승인입니다. 실제 결제에는 다음 작업이 추가로 필요합니다.

- Apple App Store 또는 Google Play 서버 영수증 검증
- 거래 ID의 전역 유일성 및 플랫폼별 서명 검증
- 환불·취소·보류·중복 영수증 처리
- 가격과 통화의 스토어 서버 동기화
- 결제 웹훅 검증과 재시도
- 개인정보·세금·미성년자 결제 등 법적 정책
- 운영자 감사, 이상 결제 탐지와 고객 지원 도구

실제 결제로 확장할 때도 플랫폼 거래 ID를 영구 저장하고, 재화 지급과 구매 원장은 PostgreSQL 트랜잭션으로 유지합니다. 클라이언트의 “결제 성공” Boolean만 신뢰해서는 안 됩니다.

## 6. 테스트

- Domain: 양수 보상 카탈로그와 골드 지급 불변식
- Application: 최초 지급, 같은 키 재생, 다른 상품 키 충돌, 원장과 이력
- API: 상품 목록, 구매, 재생, 잔액과 구매 이력
- Infrastructure: 실제 PostgreSQL에서 상태·영수증·원장 각 1건 저장
