# 영웅 강화

## 1. 구현 목표

플레이어가 보유 골드를 사용해 주 영웅을 한 레벨 올립니다. 강화 비용, 골드 차감, 성공 여부는 서버가 결정하며 네트워크 재시도와 동시 요청에도 골드가 중복 차감되거나 한 번의 골드로 여러 레벨이 오르면 안 됩니다.

## 2. 비용 규칙

현재 레벨을 `L`이라고 할 때 다음 레벨의 비용은 아래와 같습니다.

```text
floor(10 × 1.15^(L - 1))
```

| 현재 레벨 | 다음 레벨 | 비용 |
| ---: | ---: | ---: |
| 1 | 2 | 10 |
| 2 | 3 | 11 |
| 3 | 4 | 13 |
| 4 | 5 | 15 |
| 100 | 101 | 10,211,421 |

`double`의 근삿값 대신 `10 × 115^(L-1) ÷ 100^(L-1)`을 `BigInteger`로 계산한 뒤 내림합니다. 따라서 운영체제나 실행 시점이 달라도 같은 비용이 나옵니다.

골드와 비용을 `long`으로 안전하게 표현할 수 있도록 최대 레벨은 297입니다. 296에서 297로 강화하는 비용까지 `long` 범위에 들어가며 최대 레벨에서는 추가 강화를 거절합니다.

## 3. API 사용

```http
POST /api/v1/heroes/main/upgrade
Authorization: Bearer {access-token}
Idempotency-Key: upgrade-20260706-001
```

요청 Body는 없습니다. 새로운 강화에는 새로운 키를 사용하고, 응답을 받지 못해 재시도할 때는 같은 키를 사용합니다.

성공하면 `200 OK`를 반환합니다.

```json
{
  "idempotencyKey": "upgrade-20260706-001",
  "outcome": "succeeded",
  "previousLevel": 1,
  "heroLevelAfter": 2,
  "goldCost": 10,
  "goldBalanceAfter": 90,
  "processedAtUtc": "2026-07-06T01:00:00+00:00",
  "isReplay": false
}
```

골드 부족이나 최대 레벨이면 상태를 변경하지 않고 `409 Conflict`와 각각 `insufficientGold`, `maxLevelReached` 결과를 반환합니다. 이미 처리한 키면 최초 결과를 그대로 반환하고 `isReplay`만 `true`가 됩니다.

내부 저장 충돌이 세 번의 짧은 재시도 후에도 계속되면 `503 Service Unavailable`을 반환합니다. 클라이언트는 잠시 후 같은 키로 다시 요청해야 합니다.

## 4. 실패도 영수증으로 저장하는 이유

멱등성은 성공 중복만 막는 기능이 아니라 같은 요청에 같은 결과를 주는 규칙입니다.

1. 골드가 0일 때 키 `A`로 강화해 골드 부족 판정을 받습니다.
2. 이후 방치 보상으로 골드 100을 얻습니다.
3. 과거 키 `A`를 재전송하면 최초 골드 부족 결과를 재생합니다.
4. 새 키 `B`를 보내야 현재 골드 기준으로 새 강화를 판정합니다.

실패를 저장하지 않으면 네트워크 재전송이 나중에 의도하지 않은 강화로 바뀔 수 있습니다.

## 5. 동시 요청 처리

서로 다른 두 강화 요청이 같은 레벨과 골드를 동시에 읽을 수 있습니다. 두 요청 모두 메모리에서 성공을 계산하더라도 먼저 저장한 요청이 PostgreSQL `xmin`을 바꿉니다. 나중 요청은 충돌하고 최신 상태를 다시 읽어 비용과 잔액을 재판정합니다.

골드가 정확히 한 번의 비용만큼 있다면 최종 결과는 다음과 같습니다.

- 첫 요청: 성공, 골드 차감, 레벨 증가
- 두 번째 요청: 충돌 후 재조회, 골드 부족
- 최종 상태: 한 레벨만 증가하고 골드는 음수가 되지 않음

같은 키가 동시에 들어오면 `(player_id, idempotency_key)` 복합 기본키가 영수증을 하나만 허용합니다.

## 6. 계층별 역할

- Domain: 정확한 비용 계산, 잔액 확인, 골드 차감과 레벨 증가
- Application: 영수증 조회, 최초 판정 저장, 최대 3회 충돌 재시도
- Infrastructure: 강화 영수증 테이블, 복합키, 체크 제약, `xmin` 충돌 변환
- API: JWT 사용자와 멱등 키 검증, 성공·실패 HTTP 응답 변환
- Tests: 비용 경계, 상태 불변식, 실패 재생, 실제 DB 동시 요청 검증
