# Redis 도입 의사결정

## 1. 결론

현재 단일 API 인스턴스 단계에서는 Redis를 추가하지 않습니다. PostgreSQL만으로 영속 상태, 동시성 제어, 멱등 처리와 재화 감사 이력을 일관된 트랜잭션으로 처리하고 있기 때문입니다.

Redis의 첫 도입 시점은 API를 두 개 이상 실행해 Rate Limit 카운터를 공유해야 할 때입니다. 그 전까지는 운영 복잡도를 늘리지 않고, 다음 콘텐츠인 모의 상점과 구매 이력 구현을 우선합니다.

## 2. 판단 기준

새 저장소는 단순히 빠르다는 이유가 아니라 다음 조건 중 하나를 충족할 때 도입합니다.

- 여러 API 인스턴스가 같은 짧은 수명의 상태를 공유해야 한다.
- PostgreSQL 읽기 부하가 측정 결과 병목이며 캐시로 개선할 수 있다.
- TTL이 핵심 규칙인 임시 데이터가 필요하다.
- Redis 장애 시 원본 데이터 손실 없이 우회하거나 안전하게 거부할 수 있다.

## 3. 기능별 결정

| 기능 | 현재 저장소 | Redis 판단 | 이유 |
| --- | --- | --- | --- |
| 골드 잔액과 원장 | PostgreSQL | 사용하지 않음 | 재화 상태와 감사 이력을 같은 트랜잭션으로 보존해야 합니다. |
| 멱등 영수증 | PostgreSQL | 원본으로 사용하지 않음 | 재시작·TTL 만료 뒤에도 중복 지급을 막아야 합니다. |
| 장비와 진행 상태 | PostgreSQL | 사용하지 않음 | 영속 상태이며 동시 변경 검증이 필요합니다. |
| 장비 마스터 | 애플리케이션 코드 | 현재 불필요 | 데이터가 작고 배포 단위로 고정되어 네트워크 캐시가 더 비쌉니다. |
| 단일 인스턴스 Rate Limit | API 메모리 | 현재 불필요 | 현재 배포 구조에서는 정확하며 가장 단순합니다. |
| 다중 인스턴스 Rate Limit | 미구현 | 첫 도입 후보 | 모든 API가 동일한 카운터를 원자적으로 공유해야 합니다. |
| 랭킹 | 미구현 | 도입 후보 | Sorted Set이 상위 순위 및 주변 순위 조회에 적합합니다. PostgreSQL 결과가 원본이어야 합니다. |
| 이벤트 참여·세션 | 미구현 | 조건부 후보 | 짧은 TTL과 빠른 존재 확인이 실제 요구사항일 때 유용합니다. |

## 4. 첫 도입 설계

다중 인스턴스 전환 시 Redis 기반 Rate Limit을 별도 Step으로 구현합니다.

1. 키에는 환경, 정책명, 분리 기준의 해시를 포함하고 JWT나 IP 원문 노출을 피합니다.
2. 카운터 증가와 TTL 설정은 Lua script 또는 동등한 원자 연산으로 처리합니다.
3. 키 TTL은 정책 윈도우보다 약간 길게 두고 무기한 키를 만들지 않습니다.
4. 제한 수치와 윈도우는 설정으로 관리합니다.
5. 두 API 인스턴스를 대상으로 공유 한도와 TTL 만료를 통합 테스트합니다.
6. Redis 연결·거부·오류 횟수와 처리 시간을 메트릭으로 관찰합니다.

API Gateway가 이미 공유 Rate Limit을 안정적으로 제공한다면 Redis를 애플리케이션에 직접 추가하지 않는 선택도 유지합니다.

## 5. 장애 정책

Redis 장애 시 모든 기능에 같은 정책을 쓰지 않습니다.

| 기능 | 장애 동작 | 근거 |
| --- | --- | --- |
| 게스트 생성 Rate Limit | Fail closed, 503 | 익명 계정 대량 생성으로 DB가 손상되는 위험을 우선 차단합니다. |
| 인증된 상태 변경 Rate Limit | 제한된 시간만 로컬 보수적 한도로 전환, 장기 장애는 503 | 정상 이용 가능성과 악용 방지를 절충합니다. |
| 읽기 캐시 | PostgreSQL로 우회 | 원본 데이터가 남아 있으므로 정확성을 유지할 수 있습니다. |
| 랭킹 캐시 | 마지막 스냅샷 표시 또는 일시 중단 | 랭킹은 재화 지급의 직접 원본으로 사용하지 않습니다. |
| 멱등·재화 처리 | Redis와 무관하게 PostgreSQL 계속 사용 | Redis 장애가 중복 지급이나 데이터 손실로 이어지지 않게 합니다. |

장애 우회는 무제한으로 유지하지 않고 시간과 요청량에 상한을 둡니다. 복구 뒤에는 PostgreSQL 원본에서 캐시를 다시 구성합니다.

## 6. 운영과 보안

- TLS, 인증, 네트워크 접근 제한을 사용합니다.
- 애플리케이션 전용 계정은 필요한 명령만 허용합니다.
- 키에 토큰, 이메일, 원문 플레이어 식별자 같은 민감 정보를 넣지 않습니다.
- 메모리 상한과 eviction 정책을 명시합니다.
- 영속 재화는 eviction 대상 Redis에 저장하지 않습니다.
- 연결 풀, timeout, 재시도와 circuit breaker를 설정하고 재시도 폭주를 막습니다.

## 7. 도입 완료 조건

- 다중 인스턴스에서 같은 사용자의 요청이 하나의 한도를 공유합니다.
- Redis 정상·지연·단절·복구 시나리오를 자동 테스트합니다.
- 장애 정책과 운영 알림이 실제 설정 및 코드와 일치합니다.
- PostgreSQL이 재화와 멱등 처리의 최종 원본으로 유지됩니다.

## 8. 다음 재검토 시점

다음 중 하나가 발생하면 이 결정을 재검토합니다.

- API를 수평 확장한다.
- 랭킹 또는 TTL 기반 이벤트 기능을 구현한다.
- 부하 테스트에서 PostgreSQL 읽기가 병목으로 측정된다.
- API Gateway의 공유 Rate Limit 사용 여부가 결정된다.
