# 골드 변경 이력 원장

이 문서는 서버 고도화 Step 1에서 추가한 골드 변경 이력 원장의 목적, 기록 규칙, 트랜잭션 경계와 검증 방법을 설명합니다.

## 1. 왜 필요한가?

`player_game_states.gold`는 현재 잔액만 보여 줍니다. 운영 중 재화 오류나 유저 문의가 발생하면 골드를 언제, 어떤 기능에서, 얼마나 얻거나 사용했는지도 확인할 수 있어야 합니다.

원장은 다음 질문에 답하는 감사 데이터입니다.

- 방치 보상으로 얼마를 받았는가?
- 영웅 강화에 얼마를 사용했는가?
- 스테이지 성공 직전 체크포인트 정산이 얼마였는가?
- 변경 전 잔액과 변경 후 잔액이 증감량과 일치하는가?
- 같은 멱등 요청이 중복 기록되지 않았는가?

## 2. 기록 대상

실제로 골드가 증가하거나 감소한 경우만 기록합니다.

| 사유 | 원장 금액 | 기록 조건 |
| --- | ---: | --- |
| `IdleRewardClaim` | 양수 | 지급 골드가 1 이상인 방치 보상 |
| `HeroUpgrade` | 음수 | 골드를 사용한 강화 성공 |
| `StageCheckpoint` | 양수 | 스테이지 성공 전 체크포인트 지급액이 1 이상 |

즉시 보상을 다시 요청해 지급액이 0인 경우, 골드 부족 강화, 최대 레벨 강화, 스테이지 실패에는 원장을 만들지 않습니다. 이러한 요청 결과는 각 기능의 멱등 영수증에 계속 보관됩니다.

## 3. 원장 불변식

모든 원장 행은 다음 등식을 만족해야 합니다.

```text
변경 후 잔액 = 변경 전 잔액 + 증감량
```

추가 규칙은 다음과 같습니다.

- 변경 전·후 잔액은 0 이상입니다.
- 증감량은 0일 수 없습니다.
- 변경 사유는 서버가 정의한 값만 사용할 수 있습니다.
- 참조 ID는 정규화된 `Idempotency-Key`이며 최대 64자입니다.
- `(player_id, reason, reference_id)`는 유일합니다.
- 처리 시각은 서버 UTC 시각으로 저장합니다.

Domain의 `GoldLedgerEntry.Create`와 PostgreSQL Check Constraint가 같은 규칙을 이중으로 방어합니다.

## 4. 원자적 저장

게임 상태, 기능 영수증, 골드 원장은 하나의 EF Core `SaveChangesAsync`에서 함께 저장됩니다.

```text
게임 규칙 계산
    ↓
PlayerGameState 골드 변경
    ↓
기능 영수증 추가
    ↓
GoldLedgerEntry 추가
    ↓
SaveChangesAsync 한 번 호출
```

EF Core는 한 번의 저장에 포함된 변경을 PostgreSQL 트랜잭션으로 처리합니다. 따라서 세 데이터 중 하나라도 저장에 실패하면 전체가 반영되지 않습니다.

낙관적 동시성 충돌이 발생하면 변경 추적을 비우고 최신 상태와 영수증을 다시 조회합니다. 재시도에서 원장도 최신 잔액을 기준으로 다시 만들어집니다.

## 5. 테이블

`gold_ledger_entries`의 주요 열은 다음과 같습니다.

| 열 | 역할 |
| --- | --- |
| `entry_id` | 원장 행의 고유 ID |
| `player_id` | 재화 소유 플레이어 |
| `reason` | 방치 보상, 강화, 스테이지 체크포인트 구분 |
| `balance_before` | 변경 전 골드 |
| `amount` | 양수 지급 또는 음수 사용량 |
| `balance_after` | 변경 후 골드 |
| `reference_id` | 기능 요청의 멱등 키 |
| `occurred_at_utc` | 서버 처리 시각 |

현재 원장은 일반 게임 클라이언트에 노출하지 않습니다. `account_type=admin` 권한을 통과한 [관리자 조회 API](ADMIN_API.md)에서만 플레이어별 원장을 읽을 수 있습니다.

## 6. 테스트 기준

- Domain 테스트는 올바른 증가와 잘못된 잔액 등식, 0원 변경 거부를 검증합니다.
- Application 테스트는 세 유스케이스의 원장 금액과 전후 잔액을 검증합니다.
- PostgreSQL 통합 테스트는 동시 요청 후 원장 개수와 금액을 검증합니다.
- 같은 멱등 요청을 재생해도 새 원장이 생성되지 않는지 확인합니다.
